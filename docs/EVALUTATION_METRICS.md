## How We Evaluate

 We are not looking for the most complete submission — we are looking for evidence of senior-level thinking. Specifically:

| What we look at | What we want to see |
|---|---|
| System design | Do your architectural decisions make sense for a high-stakes regulatory context? Do you acknowledge trade-offs? |
| Code quality | Is the code clean, readable, and maintainable? Would another engineer be able to work in this codebase? |
| Security & correctness | Is role enforcement solid? Is the audit trail genuinely tamper-proof? Is concurrent access handled correctly? |
| Judgment under ambiguity | When the requirements didn't tell you what to do, did you make a defensible call and document it? |
| Testing | Do your tests cover the things that actually matter — business rules, security boundaries, and failure cases? |
| Communication | Can someone run and understand this system from your README and design document alone? |
